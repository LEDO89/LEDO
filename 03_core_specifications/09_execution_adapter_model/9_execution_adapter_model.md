# **Ontology Centric “Execution Adapter Model”**

# **1\. Purpose**

The Execution Adapter Model is a Core Specification that defines how an ontology-centric platform safely delivers requests to external systems after an `ApprovedAction` has been converted into an `ExecutionRequest`.

This document answers the following questions:

When does an ApprovedAction become an ExecutionRequest?  
How are ExecutionRequest and ExternalControlRequest different?  
What is the responsibility boundary of the Execution Adapter?  
Where does the responsibility of the external system begin?  
Where are the boundaries with robots, PLCs, SCADA, Fleet Managers, and Access Control Systems?  
How are physical commands separated from internal platform requests?  
How are external system requests traced, retried, and protected against duplicate execution?  
How does feedback from external systems return to the platform?  
How are failures, timeouts, network disconnection, and partial success handled?  
What happens if an external system stops between ACK and ACCEPT?  
How is time synchronization guaranteed across heterogeneous equipment?  
How is recovery defined while considering physical inertia and safety radius?  
Where is idempotency stored and verified in field Edge environments?  
How are ActionCandidates generated by SLMs or LLMs controlled by the Execution Adapter?

The core purposes of the Execution Adapter Model are:

Separate internal platform meaning from external system execution boundaries.  
Prevent ApprovedAction from becoming a direct physical command.  
Safely transform ExecutionRequest into external system requests.  
Trace responses and feedback from external execution systems.  
Manage the risk zone between ACK and ACCEPT.  
Clarify time synchronization assumptions across heterogeneous systems.  
Prevent duplicate execution.  
Make failures and timeouts recoverable states.  
Support safe-state recovery while considering physical inertia.  
Preserve an auditable execution flow.

# **3\. Core Principles**

The core principles of the Execution Adapter Model are:

ApprovedAction is not a physical command.  
ExecutionRequest is not a physical command.  
ExternalControlRequest is still a request to an external system.  
The external system owns actual physical execution.  
Feedback must return to the platform.  
Audit must preserve the full execution trace.

In plain terms:

ApprovedAction is not a physical command.  
ExecutionRequest is not a physical command.  
ExternalControlRequest is still a request to an external system.  
Actual physical execution is owned by the external system.  
Feedback from the external system must return to the platform.  
The full execution flow must be auditable.

In this model, the platform is responsible for meaning, policy, approval, Safety Gate validation, request creation, tracing, audit, and feedback collection.

External systems are responsible for actual physical execution, equipment control, robot motion, PLC logic, SCADA operation, and lock/unlock actions.

# **4\. Overall Execution Flow**

The Execution Adapter Model is positioned in the following flow:

ActionCandidate  
→ DecisionCase  
→ ApprovalRequest  
→ ApprovalDecision  
→ ApprovedAction  
→ Safety Gate  
→ ExecutionRequest  
→ Execution Adapter  
→ ExternalControlRequest  
→ External System  
→ Physical Execution  
→ FeedbackEvent  
→ World State Reconciliation  
→ AuditRecord

The important point is that the `Execution Adapter` does not directly perform physical execution.

The Execution Adapter performs the following:

Reviews the ExecutionRequest.  
Checks external system capability and request format.  
Transforms the request into an ExternalControlRequest suitable for the external system.  
Dispatches the request.  
Receives acknowledgment.  
Receives acceptance or detects acceptance timeout.  
Receives feedback or detects feedback timeout.  
Returns the result as a FeedbackEvent.  
Connects the audit trace.

Directly executing SLM or LLM outputs

The Execution Adapter is a boundary model connected to external systems.

The internal control logic for actual physical execution is owned by the external system.

# **7\. Core Object Distinctions**

## **7.1 ApprovedAction**

`ApprovedAction` is an approved action.

Examples:

ACTION\_STOP\_WORK for Zone\_A  
ACTION\_LOCK\_ZONE for Zone\_B  
ACTION\_RETURN\_ROBOT\_TO\_SAFE\_ZONE for Robot\_07

However, ApprovedAction is not yet an execution request.

ApprovedAction means:

This action has satisfied policy and approval conditions.  
However, immediate pre-execution validation is still required.

---

## **7.2 ExecutionRequest**

`ExecutionRequest` is an internal platform execution request created after passing the Safety Gate.

ExecutionRequest is an internal platform object.

Examples:

Create execution request for ACTION\_LOCK\_ZONE targeting Zone\_A.  
Create execution request for ACTION\_RETURN\_ROBOT\_TO\_SAFE\_ZONE targeting Robot\_07.

ExecutionRequest is not yet a physical command.

ExecutionRequest should be designed around references rather than directly carrying all fields.

approved\_action\_ref  
safety\_gate\_result\_ref  
execution\_context\_snapshot\_ref  
action\_type  
target\_entity\_refs  
execution\_mode  
required\_adapter\_type  
required\_capability  
idempotency\_key  
trace\_id  
decision\_trace\_id

---

## **7.3 ExecutionContextSnapshot**

`ExecutionContextSnapshot` is an object that groups execution-related context at the time an ExecutionRequest is created.

If ExecutionRequestDTO directly carries too many fields, the DTO becomes bloated, and testing and audit become difficult.

Therefore, Safety Gate results, policy decisions, evidence status, state snapshots, and time synchronization conditions should be separated into `ExecutionContextSnapshot` or reference-based objects.

ExecutionContextSnapshot may include:

execution\_context\_snapshot\_id  
safety\_gate\_result\_ref  
policy\_decision\_refs  
approval\_decision\_ref  
state\_snapshot\_ref  
evidence\_bundle\_refs  
evidence\_freshness\_status  
time\_trust\_level  
clock\_sync\_status  
clock\_drift\_estimate\_ms  
device\_health\_snapshot\_ref  
network\_health\_snapshot\_ref  
adapter\_status\_ref  
platform\_time\_ref  
created\_at  
trace\_id  
correlation\_id  
decision\_trace\_id

This structure provides the following benefits:

Keeps ExecutionRequestDTO lightweight.  
Separates audit context.  
Makes test fixture management easier.  
Separates external payload from internal judgment context.  
Strengthens connections to the Policy Governance Model and Evidence Model.

---

## **7.4 ExternalControlRequest**

`ExternalControlRequest` is a request delivered to a specific external system.

Examples:

FleetManagerRequest  
RobotMiddlewareRequest  
PLCControlRequest  
SCADAOperationRequest  
AccessControlRequest  
NotificationSystemRequest

ExternalControlRequest is closer to the API or message format of the external system.

However, from the platform perspective, it is still a “request.”

Actual physical execution is owned by the external system.

ExternalControlRequest also should not directly carry all payload fields.

external\_payload\_ref  
external\_payload\_hash  
execution\_request\_ref  
adapter\_ref  
idempotency\_key  
dispatch\_context\_ref  
trace\_id  
decision\_trace\_id

Complex vendor-specific payloads should be separated through `external_payload_ref`.

---

## **7.5 Physical Command**

A Physical Command is a command delivered inside an external system to actual equipment, robots, PLCs, SCADA, access control devices, or other physical assets.

This platform does not own Physical Commands.

The distinction is:

ApprovedAction  
→ approved action

ExecutionRequest  
→ internal platform execution request

ExternalControlRequest  
→ request delivered to an external system

Physical Command  
→ actual physical command inside the external system

# **11\. Physical Safety Capability**

In a Cyber-Physical System, physical inertia must be considered during failure handling and recovery.

Software requests can be created immediately, but moving robots, cranes, mobile equipment, or high-voltage equipment may become more dangerous if stopped abruptly.

Therefore, Adapter Capability must include physical safety capabilities beyond simple stop.

CONTROLLED\_STOP  
SAFE\_DECELERATION  
HOLD\_POSITION  
RETURN\_TO\_SAFE\_ZONE  
MAINTAIN\_SAFE\_DISTANCE  
PAUSE\_MISSION  
DISABLE\_NEW\_TASK\_ASSIGNMENT  
REQUEST\_OPERATOR\_INTERVENTION

Core principles:

Fail-safe does not always mean immediate stop.  
Fail-safe must map to the safest external-system-supported capability.  
Physical inertia and safety radius must be considered.

In plain terms:

Fail-safe does not always mean immediate stop.  
Fail-safe must be mapped to the safest capability supported by the external system.  
Physical inertia and safety radius must be considered.

For example, immediately stopping a high-speed robot may increase the risk of overturning, collision, or dropped payloads.

If the external robot system supports `SAFE_DECELERATION` or `CONTROLLED_STOP`, controlled deceleration should take priority over simple stop.

---

## **11.1 Physical Safety Capability Mapping Example**

In fail-safe situations, `action_type` must be mapped to external system capabilities.

ACTION\_STOP\_WORK  
→ HumanTaskAdapter: request field stop confirmation  
→ NotificationAdapter: send stop-work alert  
→ AccessControlAdapter: restrict access  
→ RobotMiddlewareAdapter: PAUSE\_MISSION or CONTROLLED\_STOP  
→ FleetManagerAdapter: DISABLE\_NEW\_TASK\_ASSIGNMENT

ACTION\_RETURN\_ROBOT\_TO\_SAFE\_ZONE  
→ RobotMiddlewareAdapter: RETURN\_TO\_SAFE\_ZONE  
→ FleetManagerAdapter: reassign route to safe zone

ACTION\_DISABLE\_ROBOT\_MISSION  
→ FleetManagerAdapter: PAUSE\_MISSION or DISABLE\_NEW\_TASK\_ASSIGNMENT  
→ RobotMiddlewareAdapter: HOLD\_POSITION or CONTROLLED\_STOP

ACTION\_LOCK\_ZONE  
→ AccessControlAdapter: LOCK\_ZONE  
→ NotificationAdapter: send zone access warning

ACTION\_SAFE\_STOP\_EQUIPMENT  
→ PLCAdapter: CONTROLLED\_STOP if supported  
→ PLCAdapter: SAFE\_DECELERATION if supported  
→ HumanTaskAdapter: request operator intervention if controlled stop is unavailable

The important point is that the same action may map to different physical safety capabilities depending on the target and adapter.

robot stop target  
→ CONTROLLED\_STOP or HOLD\_POSITION

crane stop target  
→ SAFE\_DECELERATION or operator-assisted controlled stop

danger zone stop-work target  
→ access restriction, notification, human task request

PLC equipment safe stop target  
→ interlock or controlled stop capability supported by the external PLC

# **13\. Time Synchronization Requirement**

The Execution Adapter Model must specify time synchronization requirements across heterogeneous systems.

`valid_until` in ExecutionRequest, `ack_deadline`, `acceptance_deadline`, `feedback_deadline` in ExternalControlRequest, and idempotency key expiry all depend on time.

Real-world risks include:

Platform server time may differ from Edge Gateway time.  
PLC or Fleet Manager internal clocks may differ.  
A few seconds of clock drift may cause incorrect timeout judgments.  
Idempotency key expiry may be interpreted differently by different systems.  
Audit log ordering may be distorted.

Therefore, time synchronization status must be validated as a prerequisite for external system integration.

Recommended requirements:

NTP or PTP synchronization should be required for production adapters.  
AdapterStatusDTO must include clock synchronization status.  
TimeSyncValidator must evaluate clock\_sync\_status, time\_trust\_level, and clock\_drift\_estimate\_ms.  
ExternalControlRequestDTO must preserve platform timestamp and adapter-local timestamp when available.  
Clock drift beyond allowed threshold must trigger revalidation or block production execution.

In plain terms:

Production Adapters must have NTP- or PTP-based time synchronization status.  
AdapterStatusDTO must include clock\_sync\_status.  
TimeSyncValidator must evaluate time\_trust\_level, clock\_sync\_status, and clock\_drift\_estimate\_ms.  
ExternalControlRequestDTO should preserve both platform timestamp and adapter-local timestamp when available.  
Clock drift beyond the allowed threshold must lead to revalidation or blocking of production execution.

# **21\. Acknowledgment and Acceptance Gap**

ACKNOWLEDGED and ACCEPTED must be clearly distinguished.

ACKNOWLEDGED  
→ external system or communication protocol layer confirmed receipt

ACCEPTED  
→ external system application layer validated the request and accepted it for execution

Real-world risks:

The request may be ACKed, but the external system application may crash.  
The network may disconnect after ACK and before ACCEPT.  
The external system may ACK the request but reject it during internal validation.  
The platform may incorrectly treat a request stuck in ACK as completed.

Therefore, a request that remains in `ACKNOWLEDGED` must be bound to `acceptance_timeout`.

Recommended rules:

ACKNOWLEDGED is not completion.  
ACKNOWLEDGED requires a maximum acceptance wait time.  
If acceptance\_timeout occurs, transition to ACCEPTANCE\_TIMEOUT.  
For high-risk requests, ACCEPTANCE\_TIMEOUT must route to fail-safe or manual override.  
For emergency requests, ACCEPTANCE\_TIMEOUT must evaluate emergency fail-safe policy.

---

## **21.1 ACK vs ACCEPT Scenario 1: External Application Crash After ACK**

Situation:

ExternalControlRequest is dispatched to FleetManager.  
Transport layer returns ACKNOWLEDGED.  
FleetManager application crashes before ACCEPTED.  
No ACCEPTED response arrives before acceptance\_deadline.

Result:

dispatch\_status → ACCEPTANCE\_TIMEOUT  
RetryPolicy must evaluate retry eligibility.  
Policy Governance Model must check whether retry is allowed.  
High-risk action may require manual override or fail-safe.  
Audit event must be created.

Core judgment:

ACK only means the external system received the request.  
ACK does not mean the external system accepted the request for execution.

---

## **21.2 ACK vs ACCEPT Scenario 2: Network Disconnection After ACK**

Situation:

AccessControlAdapter sends request to lock Zone\_A.  
External system returns ACKNOWLEDGED.  
Network disconnects before ACCEPTED or REJECTED is received.

Result:

ExternalControlRequest enters ACCEPTANCE\_PENDING.  
acceptance\_timeout expires.  
Grace Period policy is evaluated if local edge policy exists.  
Duplicate retry is blocked unless idempotency guard allows safe retry.  
Manual inspection may be required.

Core judgment:

Network disconnection does not expand authority.  
Duplicate dispatch must be blocked by the idempotency guard.

---

## **21.3 ACK vs ACCEPT Scenario 3: External Validation Rejection After ACK**

Situation:

RobotMiddleware receives request.  
Transport layer returns ACKNOWLEDGED.  
Application layer validates target robot state.  
Robot is already in maintenance mode.  
External system returns REJECTED.

Result:

dispatch\_status → REJECTED  
FailureCode → EXTERNAL\_REJECTED  
World State Reconciliation may be required.  
Policy review may be required if platform state was outdated.

Core judgment:

External system REJECTED is not merely a failure.  
It is important state feedback.  
The platform must connect it to Evidence and State Reconciliation.

---

# **22\. Acknowledgment Model**

Acknowledgment is the signal that the external system received the request.

Acknowledgment does not mean success.

ACKNOWLEDGED  
→ request received

ACCEPTANCE\_PENDING  
→ waiting for application-level acceptance

ACCEPTED  
→ external system accepted the request for processing

COMPLETED  
→ actual processing completed

FAILED  
→ processing failed

This distinction must remain clear.

For example, an external system may receive a request but reject execution due to internal conditions.

# **24\. Idempotency**

Idempotency is critical in the Execution Adapter Model.

Even if network retries, duplicate messages, repeated user requests, or delayed external system responses occur, the same ExecutionRequest must not be executed more than once.

Idempotency criteria:

ExecutionRequest must have idempotency\_key.  
ExternalControlRequest must preserve idempotency\_key.  
Retry must reuse the same idempotency\_key.  
Duplicate dispatch must be detected.  
External feedback must be matched by idempotency\_key or external\_reference\_id.

---

# **25\. Distributed Idempotency Guard**

In CPS environments, storing the idempotency key only in the central database is insufficient.

Real-world risks:

The network between the central server and Edge Adapter may disconnect.  
The Adapter may need to determine retry eligibility locally.  
The same ExternalControlRequest may be dispatched more than once.  
External system feedback may arrive late.

Therefore, the IdempotencyGuard should also exist close to the execution boundary.

Recommended structure:

Central Idempotency Store  
→ global execution history and audit reference

Edge Idempotency Guard  
→ prevents duplicate dispatch at the field Edge or Adapter boundary

External System Idempotency  
→ connects to external references if supported by the external system

Implementation examples:

central database  
edge local key-value store  
Redis-compatible cache  
embedded local store  
adapter-local idempotency ledger

The IdempotencyGuard must manage:

idempotency\_key  
idempotency\_scope  
execution\_request\_ref  
external\_control\_request\_ref  
first\_seen\_at  
last\_seen\_at  
expires\_at  
dispatch\_count  
last\_dispatch\_status  
external\_reference\_id  
trace\_id  
decision\_trace\_id

Core principles:

Idempotency must work even during partial network failure.  
Retry must not create duplicate physical execution.  
Edge Adapter must be able to detect duplicate dispatch locally.

---

# **26\. Retry Policy**

Retry must not be repeated unconditionally.

Retry must be restricted according to action type, adapter type, risk level, external system status, and the Policy Governance Model decision result.

Recommended retry evaluation conditions:

Retry result values:

RETRY\_ALLOWED  
RETRY\_DENIED  
RETRY\_DELAYED  
RETRY\_REQUIRES\_MANUAL\_REVIEW  
RETRY\_REQUIRES\_FAIL\_SAFE

Retry must always use the same idempotency\_key.

When Grace Period is active, retry authority must not be expanded.

Communication loss is not a condition that expands authority; it is a condition that limits authority.

Retry for emergency requests must follow both Emergency Policy and Post-Hoc Audit conditions.

# **28\. Recovery Model**

The Execution Adapter Model must define recovery flows after failure.

Recovery is not simple rollback.

In particular, rollback can be dangerous in a Cyber-Physical System.

Recommended Recovery results:

NO\_RECOVERY\_REQUIRED  
RETRY\_REQUIRED  
STATE\_RECONCILIATION\_REQUIRED  
CONTROLLED\_STOP\_REQUIRED  
HOLD\_POSITION\_REQUIRED  
FAIL\_SAFE\_REQUIRED  
MANUAL\_OVERRIDE\_REQUIRED  
POLICY\_REVIEW\_REQUIRED  
ADAPTER\_REPAIR\_REQUIRED  
AUDIT\_REVIEW\_REQUIRED

Core principles:

Cyber-physical recovery is not database rollback.  
Cyber-physical recovery means safe-state transition, reconciliation, manual review, or fail-safe action.  
Physical inertia must be considered.  
Immediate stop is not always the safest recovery.

In plain terms:

CPS recovery is not database rollback.  
CPS recovery means safe-state transition, state reconciliation, human review, or fail-safe action.  
Physical inertia must be considered.  
Immediate stop is not always the safest recovery.

For example, moving robots, cranes, or high-load lift equipment may be safer under `CONTROLLED_STOP`, `SAFE_DECELERATION`, `HOLD_POSITION`, or `RETURN_TO_SAFE_ZONE` than under immediate stop.

The Execution Adapter must check which capabilities are supported by the external system when creating recovery requests.

# **29\. SLM / LLM ActionCandidate Execution Boundary**

The Execution Adapter must not directly execute outputs generated by SLMs or LLMs.

However, SLMs or LLMs must not generate:

ApprovedAction

An ActionCandidate generated by an SLM must pass through the following route:

→ ActionCandidate  
→ Policy Governance Check  
→ Decision / Approval Matrix  
→ Approval if required  
→ Safety Gate  
→ ExecutionRequest  
→ Execution Adapter

The Execution Adapter must not process an SLM-originated candidate unless all of the following exist:

approved\_action\_ref exists  
safety\_gate\_result\_ref exists  
execution\_context\_snapshot\_ref exists  
ai\_governance\_policy\_ref exists  
trace\_id exists  
decision\_trace\_id exists

# **37\. Final Principles**

The Execution Adapter Model is the boundary model between the platform and external physical execution systems.

This model does not own execution.

This model manages execution request meaning, conditions, tracing, duplicate execution prevention, and feedback collection.

Core principles:

ApprovedAction is not a physical command.  
ExecutionRequest is not a physical command.  
ExternalControlRequest is still a request.  
Physical execution belongs to external systems.  
ACK is not ACCEPT.  
ACCEPT is not COMPLETE.  
Adapter must preserve idempotency.  
Adapter must support local idempotency guarding when deployed at the edge.  
Adapter timeout policy must handle ACK, ACCEPT, execution, feedback, and reconciliation separately.  
Adapter must validate time synchronization for production execution.  
Adapter must reference Evidence Model fields for time trust, clock sync, clock drift, and device health.  
Adapter must not create ExternalControlRequest from stale high-risk evidence.  
Adapter must map fail-safe to external-system-supported physical safety capabilities.  
Adapter must not directly execute SLM or LLM output.  
Adapter must return feedback.  
Adapter failure must be traceable.  
Recovery means safe-state handling, not simple rollback.  
Audit must preserve the full execution chain.
