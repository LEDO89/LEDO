# **Ontology foundation “Reasoning and Constraint” Model** 

## **1\. Purpose**

This document defines the responsibilities, execution locations, materialization methods, and runtime boundaries of reasoning and constraint validation in the LEDO Industrial Ontology Foundation.

In LEDO, reasoning is not a single technology. OWL Reasoning, SHACL Validation, SPARQL Graph Validation, Policy Evaluation, Solver Optimization, and Safety Gate Snapshot Lookup each have different responsibilities.

The purpose of this document is to clarify the following.

What does OWL reasoning handle?  
What does SHACL constraint validation check?  
What relationship conditions are checked by SPARQL / graph validation?  
What operational decisions are performed by the Policy Engine?  
What optimization candidates are generated by the Solver?  
What does the Safety Gate lookup instead of newly computing?  
How are reasoning results converted into Materialized Views or Safety Gate Runtime Snapshots?  
What operations are prohibited in the Safety Gate hot path?

The core principle is as follows.

OWL infers meaning.  
SHACL validates data shape.  
SPARQL verifies relationship conditions.  
Policy determines operational permission.  
Solver computes optimization candidates.  
Safety Gate only looks up precomputed results.

## **3\. Core Problem**

In an industrial Physical AI system, the following questions arise at the same time.

Which class does this object belong to?  
Is this relation possible according to the ontology?  
Does this data contain the required fields?  
Can this observation be used as evidence?  
Is this ActionCandidate semantically compatible with its target class?  
Is this action allowed under the current policy?  
Does this action require approval?  
Does this action conflict with the current world state?  
Can this action be delivered to an external control system?

These questions must not be delegated to a single reasoner, a single rule engine, or a single LLM.

Each question must be handled by a different layer.

| Question Type | Responsible Layer |
| ----- | ----- |
| Meaning question | OWL / Ontology |
| Data shape question | SHACL / JSON Schema / DTO Validator |
| Graph relationship question | SPARQL / Graph Validation |
| Permission and policy question | Policy Engine |
| Optimization question | Solver |
| Pre-execution readiness question | Safety Gate Runtime Snapshot |
| Situation interpretation question | Agent SLM / Domain Agent |

---

## **4\. Reasoning and Constraint Responsibility Matrix**

In LEDO, the responsibilities of reasoning and constraint validation are separated as follows.

| Category | Main Purpose | Technology | Runtime Location |
| ----- | ----- | ----- | ----- |
| Semantic Reasoning | class membership, subclass, property inference | OWL, RDFS, HermiT, Pellet, OWLReady2 | offline / controlled background |
| Shape Validation | required field, datatype, DTO shape validation | SHACL, JSON Schema, Pydantic | ingestion / candidate validation |
| Graph Relationship Validation | risk-zone-evidence-action relationship check | SPARQL ASK, Graph Query, Cypher | candidate validation / materialization |
| Policy Evaluation | authority, approval, operational permission | OPA/Rego, RBAC, ABAC, Approval Matrix | pre-check / materialization |
| Temporal Validation | freshness, validity interval, timestamp ordering | OWL-Time, timestamp logic, time-series query | pre-check / snapshot |
| Spatial Validation | zone, boundary, containment, geofence check | GeoSPARQL, spatial index, zone map | pre-check / snapshot |
| Solver Optimization | route, allocation, resource assignment candidate | OR-Tools, Z3, route planner | async worker |
| Safety Gate Lookup | final pre-execution validation | immutable snapshot, bitset, enum, flat map | hot path |
| Agent Interpretation | situation interpretation and candidate generation | SLM, LoRA, TinyML, Vision Model | domain agent layer |

The core principle is as follows.

Reasoning must be separated according to runtime location.  
Heavy reasoning must be performed outside the hot path.  
The Safety Gate must not newly perform reasoning; it must only look up results.

## **5\. Runtime Positioning Model**

LEDO divides reasoning into three runtime zones.

### **5.1 Offline / Controlled Reasoning**

Heavy reasoning is performed offline or inside a controlled background worker.

Included tasks are:

OWL reasoner execution  
TBox consistency check  
ABox sample validation  
Large SPARQL query  
GeoSPARQL spatial relation expansion  
Solver optimization  
Policy matrix precomputation  
Snapshot generation

The results of this stage are converted into:

Materialized Semantic View  
Materialized Graph View  
Policy Decision Table  
Safety Gate Runtime Snapshot

---

### **5.2 Runtime Pre-check Reasoning**

This is validation performed after an ActionCandidate is created and before it reaches the Decision Router.

Its position is as follows.

ActionCandidate  
→ Candidate Semantic Validation  
→ Shape Validation  
→ Relation Graph Validation  
→ Evidence Sufficiency Check  
→ Policy Pre-check  
→ Decision Router

In this zone, SHACL, SPARQL, and policy checks may be performed.  
However, this stage must occur before the Safety Gate hot path.

---

### **5.3 Safety Gate Hot Path**

The Safety Gate hot path is the final checkpoint immediately before execution.

This zone does not perform new reasoning.

The following operations are prohibited.

No OWL reasoner  
No full SHACL validation  
No SPARQL query  
No Graph DB network call  
No LLM / SLM call  
No disk I/O  
No external API call  
No solver execution

The following operations are allowed.

Read active snapshot  
Check bitset  
Check enum  
Check timestamp freshness flag  
Check approval flag  
Check external system health flag  
Check conflict matrix  
Return allow / deny / hold / escalate

The core principle is as follows.

The Safety Gate hot path is not a reasoning layer.  
The Safety Gate hot path is a deterministic snapshot lookup layer.

---

## **6\. Candidate Validation Flow**

The validation flow after an ActionCandidate is created is as follows.

1\. Domain Agent  
   → Interprets the situation and creates an ActionCandidate

2\. Candidate Semantic Validation  
   → Validates action type, target class, and ontology meaning

3\. Shape Validation  
   → Validates DTO fields, datatypes, and required properties

4\. Relation Graph Validation  
   → Validates risk-zone-evidence-action relationships

5\. Evidence Sufficiency Check  
   → Validates evidence existence, provenance, and freshness

6\. Policy Pre-check  
   → Validates authority, approval requirements, and operational policy

7\. Decision Router  
   → Determines risk tier, urgency, and approval route

8\. Approval  
   → Handles human/system approval

9\. Safety Gate  
   → Performs final pre-execution validation based on the snapshot

10\. ExecutionRequest  
   → Creates a high-level request that can be delivered to the external execution layer

The responsibilities of each step are as follows.

| Step | Responsibility |
| ----- | ----- |
| Domain Agent | situation interpretation and candidate generation |
| Candidate Semantic Validation | ontology meaning, target class, action type validation |
| Shape Validation | DTO structure, required fields, datatype validation |
| Relation Graph Validation | graph relation consistency validation |
| Evidence Sufficiency Check | evidence existence, provenance, freshness check |
| Policy Pre-check | authority, approval requirements, operational policy check |
| Decision Router | risk, urgency, approval tier classification |
| Approval | human/system approval handling |
| Safety Gate | final pre-execution validation using snapshot |
| ExecutionRequest | high-level request delivered to external execution layer |

The core boundary is as follows.

An ActionCandidate is not an execution command.  
An ApprovedAction is not a physical command.  
An ExecutionRequest is not a low-level control command.  
The external controller / PLC / robot middleware performs actual physical control.

## **7\. Reasoning Result Materialization**

Reasoning results are not computed directly inside the Safety Gate hot path.

Instead, they are materialized as follows.

OWL Reasoning  
→ inferred\_class\_map  
→ class\_membership\_bitset

SHACL Validation  
→ shape\_validity\_flag

SPARQL / Graph Validation  
→ relation\_allowed\_map

Policy Evaluation  
→ action\_permission\_map

Temporal Validation  
→ evidence\_freshness\_map  
→ permit\_validity\_map

Spatial Validation  
→ zone\_restriction\_map  
→ geofence\_violation\_map

External System Check  
→ external\_system\_health\_map

Conflict Check  
→ conflict\_matrix

Ultimately, these results are stored in the Safety Gate Runtime Snapshot.

The core principle is as follows.

Reasoning runs in the background.  
Validation results are converted into materialized maps.  
The Safety Gate looks up materialized results.

---

## **8\. Safety Gate Runtime Snapshot Structure**

The Safety Gate Runtime Snapshot is an immutable runtime view read by the Safety Gate in the hot path.

The basic structure is as follows.

SafetyGateSnapshot  
  version  
  generated\_at  
  valid\_until  
  ontology\_version  
  policy\_version  
  shacl\_shape\_version  
  inference\_version  
  checksum

  object\_state\_map  
  action\_permission\_map  
  risk\_action\_matrix  
  zone\_restriction\_map  
  evidence\_freshness\_map  
  approval\_state\_map  
  external\_system\_health\_map  
  conflict\_matrix  
  inferred\_class\_map  
  relation\_allowed\_map

This snapshot must satisfy the following conditions.

immutable  
read-only  
versioned  
checksum-verified  
precomputed  
hot path safe  
fail-closed compatible

The following is a conceptual example.

  inference\_version: "inference-v0.6.3"  
  generated\_at: "2026-06-25T09:00:00Z"  
  valid\_until: "2026-06-25T09:00:05Z"  
  checksum: "sha256:..."

  evidence\_freshness\_map:  
    Evidence\_GasSensor\_01\_0900 → fresh  
    Evidence\_GasSensor\_01\_0830 → stale

  action\_permission\_map:  
    RobotAgent\_01 \+ restrict\_access \+ Zone\_A → allowed  
    RobotAgent\_01 \+ direct\_physical\_control \+ Valve\_01 → denied  
    SafetySupervisor\_01 \+ approve\_high\_risk\_action \+ Zone\_A → allowed

  zone\_restriction\_map:  
    Zone\_A → restricted  
    Zone\_B → speed\_limited  
    Zone\_C → normal

  approval\_state\_map:  
    Approval\_1001 → valid  
    Approval\_1002 → expired

  external\_system\_health\_map:  
    PLC\_01 → available  
    RobotMiddleware\_02 → unavailable  
    VentilationController\_03 → available

  conflict\_matrix:  
    dispatch\_robot \+ locked\_zone → conflict  
    notify\_manager \+ restricted\_zone → no\_conflict

## **10\. Temporal Reasoning Materialization**

Temporal reasoning handles freshness, validity, and ordering.

Representative questions include:

Is this evidence still fresh?  
Is this permit currently valid?  
Has this approval not yet expired?  
During which time interval is this risk state valid?  
Was this observation created before the action candidate?

Examples of materialized maps are:

evidence\_freshness\_map:  
  Evidence\_GasSensor\_01\_0900 → fresh  
  Evidence\_GasSensor\_01\_0830 → stale

permit\_validity\_map:  
  Permit\_Work\_77 → valid  
  Permit\_Work\_12 → expired

approval\_state\_map:  
  Approval\_1001 → valid  
  Approval\_1002 → expired

risk\_state\_validity\_map:  
  GasRisk\_Zone\_A → valid\_until 2026-06-25T09:05:00Z

The Safety Gate does not newly infer time.  
It only looks up precomputed freshness flags and validity flags.

---

## **11\. Spatial Reasoning Materialization**

Spatial reasoning handles zones, boundaries, containment, geofences, and speed-limited areas.

Representative questions include:

Is Agent\_01 inside RestrictedZone?  
Is Zone\_A contained within GeofencingBoundary\_B?  
Which physical area does SpeedLimitedZone apply to?  
Which spatial region is affected by the RiskEvent?

Examples of materialized maps are:

zone\_containment\_map:  
  RobotAgent\_01 → inside Zone\_A  
  Worker\_07 → outside RestrictedZone\_B

zone\_restriction\_map:  
  Zone\_A → restricted  
  Zone\_B → speed\_limited  
  Zone\_C → normal

geofence\_violation\_map:  
  RobotAgent\_01 \+ Zone\_A → no\_violation  
  RobotAgent\_02 \+ RestrictedZone\_B → violation

speed\_limit\_zone\_map:  
  Zone\_B → max\_speed 0.5m/s  
  Zone\_C → no\_speed\_limit

GeoSPARQL queries are not executed directly in the Safety Gate hot path.  
GeoSPARQL or spatial index results must be converted into materialized maps in advance.

---

## **12\. Separation Between Open World and Closed World**

OWL follows the Open World Assumption.

Lack of information does not mean false.

However, runtime validation is closer to a closed-world model.

If required information is missing, it is invalid.  
If required evidence is missing, hold or deny.  
If required approval is missing, deny.  
If no snapshot exists, do not allow.

Summary:

OWL  
→ unknown is possible

SHACL / Policy / Safety Gate  
→ missing required data is a violation or denial condition

The core principle is as follows.

Do not confuse open-world reasoning with runtime validation.

## **13\. Solver and PlanCandidate**

The Solver computes optimization candidates.

Allowed use cases include:

Task allocation  
Route planning  
Resource optimization  
Schedule adjustment  
Conflict resolution proposal

A Solver result is not an execution command.  
A Solver result is a `PlanCandidate`.

PlanCandidate  
→ Candidate allocation, route, schedule, or resource assignment computed by the solver  
→ Operational proposal  
→ Not an ApprovedAction  
→ Not an ExecutionRequest  
→ Not a physical command

A PlanCandidate must go through the following flow.

Solver Result  
→ PlanCandidate  
→ Semantic Validation  
→ Policy Check  
→ Approval  
→ Safety Gate  
→ ExecutionRequest

The core principle is as follows.

The Solver creates a candidate plan.  
The Solver has no physical execution authority.

---

## **14\. Boundary Between AI Output and Reasoning**

AI output is not a reasoning result.

SLM / LLM output  
→ interpretation  
→ candidate  
→ hypothesis

AI output cannot directly become:

Evidence  
ApprovedAction  
ExecutionRequest  
Safety Gate decision  
Ontology fact  
Policy decision

For AI output to be reflected in the ontology or ABox, it must go through the following flow.

AI output  
→ Candidate  
→ Evidence Binding  
→ Validation  
→ Human or Policy Review  
→ Ontology / ABox update  
→ Audit

The core principle is as follows.

AI creates candidates.  
The ontology and validation layers validate those candidates.

---

## **15\. Anti-Patterns**

| Anti-pattern | Problem | Alternative |
| ----- | ----- | ----- |
| Running the OWL reasoner in the Safety Gate hot path | Runtime jitter occurs | Reasoning result materialization |
| Running full SHACL validation in the Safety Gate hot path | Execution delay occurs | Shape result flag lookup |
| Running SPARQL queries in the hot path | Triple store latency occurs | Materialized relation map |
| Replacing Policy with OWL axioms | Operational policy becomes difficult to change | Separate Policy Engine |
| Using AI output as Evidence | Hallucination is treated as fact | Evidence binding required |
| Using Solver result directly as ExecutionRequest | Validation/approval is bypassed | Treat as PlanCandidate |
| Trying to make OWL invalidate missing data | Conflicts with Open World | Use SHACL / DTO validation |
| Calling external APIs inside the Safety Gate | Network jitter occurs | Use health state snapshot |
| Modeling runtime state as a TBox class | Time change cannot be represented | Use State/Event/TimeInterval |
| Recomputing reasoning results every time without materialization | Tail latency increases | Snapshot precomputation |

---

## **16\. Final Operating Principles**

Semantic reasoning belongs to OWL.  
Data validation belongs to SHACL.  
Graph relationship validation belongs to SPARQL / graph query.  
Operational permission belongs to Policy Engine.  
Optimization belongs to Solver.  
Runtime execution readiness belongs to Safety Gate Runtime Snapshot.  
AI output is candidate interpretation.  
The Safety Gate hot path must only look up precomputed results.

---

## **17\. Final Conclusion**

The core of the LEDO Reasoning and Constraint Model is not to add more reasoning everywhere.

The core is to separate reasoning responsibilities precisely, materialize heavy reasoning results before runtime, and allow the Safety Gate to perform deterministic lookup.

The final structure is as follows.

Ontology / OWL  
→ meaning and inference

SHACL  
→ data shape validation

SPARQL / Graph Query  
→ relationship validation

Policy Engine  
→ permission and approval logic

Solver  
→ optimization candidate

World State  
→ current runtime facts

Safety Gate Runtime Snapshot  
→ precomputed execution readiness view

Safety Gate  
→ deterministic final lookup

The final principle is as follows.

Reasoning creates meaning.  
Constraint validation filters invalid structure.  
Policy determines permission.  
Snapshot materializes runtime-ready facts.  
Safety Gate only reads and decides.

Reasoning 결과는 어떻게 materialized view 또는 Safety Snapshot으로 변환되는가?

OWL은 의미를 추론한다.

## **4\. Reasoning 종류**

LEDO는 reasoning을 다음과 같이 구분한다.

### **4.1 Semantic Reasoning**

Semantic reasoning은 ontology meaning을 기반으로 class membership, subclass hierarchy, property relation을 추론한다.

담당 기술:

OWL  
RDFS  
OWL Reasoner  
HermiT  
Pellet  
OWLReady2

예시:

GasSensor subClassOf Sensor.  
Sensor subClassOf PhysicalObject.

GasSensor\_01 rdf:type GasSensor.

Reasoning result:  
GasSensor\_01 rdf:type Sensor.  
GasSensor\_01 rdf:type PhysicalObject.

사용 위치:

Ontology validation  
TBox consistency check  
ABox sample validation  
Class hierarchy inference  
Materialized semantic view generation  
Safety Snapshot precomputation

금지 위치:

Safety Gate hot path  
Real-time execution validation  
Low-level physical control loop

---

### **4.2 Shape / Data Constraint Validation**

Shape validation은 실제 데이터가 요구된 구조를 갖추었는지 검증한다.

담당 기술:

SHACL  
JSON Schema  
Pydantic  
DTO Validator

예시:

SensorObservation must have generatedBy.  
generatedBy must point to a Sensor.  
timestamp must exist.  
confidence must be a number.

사용 위치:

Data ingestion  
ABox creation  
Canonical object creation  
ActionCandidate validation  
EvidenceBundle validation  
Synthetic data filtering

핵심 원칙:

OWL restriction은 의미 조건이다.  
SHACL shape는 실제 데이터 검증이다.

---

### **4.3 Graph Relationship Validation**

Graph validation은 특정 관계 조건이 graph 안에서 성립하는지 확인한다.

담당 기술:

SPARQL ASK  
SPARQL SELECT  
Graph query  
Neo4j / Cypher  
Materialized graph view

예시 질문:

이 ActionCandidate의 target은 Zone인가?  
이 RiskEvent는 해당 Zone에 영향을 주는가?  
이 EvidenceBundle은 해당 Observation에서 생성되었는가?  
이 Sensor는 해당 Zone을 observe하는가?  
이 action type은 해당 risk type을 mitigate할 수 있는가?

사용 위치:

Candidate Semantic Validation  
Relation Graph Validation  
Evidence Sufficiency Check  
Audit Query  
Materialized View Generation

금지 위치:

Safety Gate hot path에서 직접 SPARQL 실행 금지

Safety Gate에서는 SPARQL 결과를 미리 materialized table로 변환해서 조회한다.

---

### **4.4 Policy Reasoning**

Policy reasoning은 운영 허용 여부를 판단한다.

담당 기술:

OPA / Rego  
Policy Engine  
RBAC  
ABAC  
Decision Table  
Approval Matrix

예시 질문:

이 사용자는 이 action을 승인할 권한이 있는가?  
이 action은 human approval을 요구하는가?  
이 zone은 현재 restricted 상태인가?  
이 action은 emergency mode에서 자동 허용되는가?  
이 role은 이 target에 대해 이 action을 요청할 수 있는가?

Policy는 ontology meaning을 참고할 수 있지만, ontology가 policy decision을 대신하지 않는다.

핵심 원칙:

OWL은 의미를 정의한다.  
Policy는 운영 허용 여부를 판단한다.

---

### **4.5 Temporal Reasoning**

Temporal reasoning은 시간 조건을 다룬다.

담당 기술:

OWL-Time  
timestamp validation  
validDuring  
event interval  
freshness window  
time-series query

예시 질문:

이 evidence는 아직 fresh한가?  
이 permit은 현재 유효한가?  
이 risk state는 언제부터 언제까지 유효한가?  
이 observation은 action candidate보다 먼저 생성되었는가?

실시간 freshness 판단은 Safety Gate Snapshot에 materialized되어야 한다.

예시:

evidence\_freshness\_map\[evidence\_id\] \= fresh / stale  
permit\_validity\_map\[permit\_id\] \= valid / expired

---

### **4.6 Spatial Reasoning**

Spatial reasoning은 위치, zone, boundary, containment, geofence를 다룬다.

담당 기술:

GeoSPARQL  
Spatial index  
Zone containment map  
Geofencing boundary  
Materialized spatial relation table

예시 질문:

Agent\_01은 RestrictedZone 안에 있는가?  
Zone\_A는 GeofencingBoundary\_B 안에 포함되는가?  
SpeedLimitedZone은 어떤 physical area에 적용되는가?  
RiskEvent가 영향을 주는 spatial region은 어디인가?

실시간 hot path에서는 GeoSPARQL query를 직접 실행하지 않는다.

대신 다음처럼 materialized한다.

zone\_containment\_map  
zone\_restriction\_map  
geofence\_violation\_map  
speed\_limit\_zone\_map

---

### **4.7 Optimization / Solver Reasoning**

Optimization reasoning은 여러 제약 조건 안에서 최적 배치, 경로, 할당을 계산한다.

담당 기술:

OR-Tools  
Z3  
Constraint Solver  
Route Planner  
Task Allocator

예시 질문:

어떤 RobotAgent가 어떤 InspectionTask에 배정되어야 하는가?  
어떤 경로가 restricted zone을 피하면서 가장 안전한가?  
어떤 resource allocation이 schedule delay를 최소화하는가?

Solver는 비동기 worker에서 실행한다.

금지 원칙:

Solver는 Safety Gate hot path에서 실행하지 않는다.  
BDI loop를 block하지 않는다.  
결과는 candidate plan 또는 recommendation으로만 사용한다.

---

### **4.8 Agent Interpretation**

Agent interpretation은 SLM/LLM이 상황을 해석하고 ActionCandidate를 생성하는 단계다.

담당 기술:

SLM  
LoRA  
TinyML  
Vision model  
BDI-lite  
Domain Agent

하지만 AI output은 truth가 아니다.

AI output  
→ candidate interpretation

Agent output  
→ ActionCandidate

ActionCandidate  
→ 검증 전 후보

핵심 원칙:

Agent는 후보를 생성한다.  
Reasoning / Validation / Policy / Safety Gate가 후보를 검증한다.

## **5\. Constraint 종류**

LEDO의 constraint는 다음과 같이 분류한다.

### **5.1 Semantic Constraint**

Ontology 의미에 관한 제약이다.

예시:

RobotAgent is a type of Agent.  
GasSensor is a type of Sensor.  
EvidenceBundle is an InformationArtifact.  
ActionCandidate must have a target Entity.

담당:

OWL  
RDFS  
TBox axiom  
Reasoner

---

### **5.2 Shape Constraint**

데이터 구조에 관한 제약이다.

예시:

Observation must have timestamp.  
EvidenceBundle must have source\_system.  
ActionCandidate must have action\_type.  
ExecutionRequest must have approved\_action\_id.

담당:

SHACL  
JSON Schema  
Pydantic  
DTO Validator

---

### **5.3 Relationship Constraint**

그래프 관계에 관한 제약이다.

예시:

GasRisk must affect some Zone.  
EvidenceBundle must support an ActionCandidate.  
SensorObservation must be generatedBy a Sensor.  
ActionCandidate target must be compatible with action type.

담당:

SPARQL ASK  
Graph query  
Materialized graph view

---

### **5.4 Policy Constraint**

운영 정책과 권한에 관한 제약이다.

예시:

High-risk action requires supervisor approval.  
RestrictedZone access requires permit.  
Robot dispatch requires capability verification.  
Emergency stop may bypass normal approval but must be audited.

담당:

OPA / Rego  
Policy Engine  
Approval Matrix  
RBAC / ABAC

---

### **5.5 Runtime State Constraint**

현재 상태에 관한 제약이다.

예시:

External system must be available.  
Evidence must be fresh.  
Target zone must not be locked.  
Robot battery must be above threshold.  
Sensor state must be healthy.

담당:

World State Cache  
Safety Gate Runtime Snapshot  
Redis  
In-memory state store

---

### **5.6 Execution Constraint**

실행 직전 조건에 관한 제약이다.

예시:

ApprovedAction must exist.  
Approval must still be valid.  
No conflicting action may be active.  
External adapter must be healthy.  
Current state must match expected precondition.

담당:

Safety Gate  
Safety Snapshot  
Execution Adapter Pre-check

---

## **6\. Reasoning 위치 원칙**

LEDO는 reasoning을 runtime 위치에 따라 구분한다.

### **6.1 Offline / Controlled Reasoning**

무거운 reasoning은 offline 또는 controlled background worker에서 수행한다.

포함:

OWL reasoner  
TBox consistency check  
ABox sample validation  
Large SPARQL query  
GeoSPARQL spatial relation expansion  
Solver optimization

결과는 다음으로 변환된다.

Materialized Semantic View  
Materialized Graph View  
Safety Gate Runtime Snapshot

---

### **6.2 Runtime Pre-check Reasoning**

ActionCandidate 이후 Decision Router 전까지 수행되는 검증이다.

위치:

ActionCandidate  
→ Candidate Semantic Validation  
→ Relation Graph Validation  
→ Evidence Sufficiency Check  
→ Policy Pre-check  
→ Decision Router

이 단계에서는 SPARQL, SHACL, policy check가 가능하지만, Safety Gate hot path보다는 앞에 있어야 한다.

---

### **6.3 Safety Gate Hot Path**

Safety Gate hot path는 reasoning을 새로 수행하지 않는다.

금지:

No OWL reasoner  
No full SHACL validation  
No SPARQL query  
No Graph DB network call  
No LLM / SLM call  
No disk I/O  
No external API call

허용:

Read active snapshot  
Check bitset  
Check enum  
Check timestamp freshness flag  
Check approval flag  
Check external system health flag  
Check conflict matrix  
Return allow / deny / hold / escalate

---

## **7\. Candidate Validation Flow**

ActionCandidate가 생성된 뒤의 검증 흐름은 다음과 같다.

Domain Agent  
→ ActionCandidate  
→ Candidate Semantic Validation  
→ Shape Validation  
→ Relation Graph Validation  
→ Evidence Sufficiency Check  
→ Policy Pre-check  
→ Decision Router  
→ Approval  
→ Safety Gate  
→ ExecutionRequest

각 단계의 책임은 다음과 같다.

Candidate Semantic Validation  
→ action type, target class, ontology meaning 검증

Shape Validation  
→ required field, datatype, DTO shape 검증

Relation Graph Validation  
→ graph relation consistency 검증

Evidence Sufficiency Check  
→ evidence 존재, provenance, freshness 검증

Policy Pre-check  
→ permission, approval requirement, role-based rule 검증

Decision Router  
→ risk, urgency, approval tier, escalation 분류

Safety Gate  
→ 실행 직전 snapshot 기반 최종 검증

---

## **8\. Reasoning Result Materialization**

Reasoning 결과는 Safety Gate hot path에서 직접 계산하지 않는다.

대신 다음 구조로 변환한다.

SafetyGateSnapshot

  generated\_at

  policy\_version  
  shacl\_shape\_version  
  inference\_version  
  checksum

  object\_state\_map  
  action\_permission\_map

핵심 원칙:

Reasoning은 background에서 수행한다.  
Safety Gate는 reasoning result를 lookup한다.

## **9\. Open World와 Closed World의 분리**

OWL은 Open World Assumption을 따른다.

정보가 없다는 것은 거짓을 의미하지 않는다.

하지만 runtime validation은 closed-world에 가깝다.

필수 정보가 없으면 invalid.  
필수 evidence가 없으면 hold 또는 deny.  
필수 approval이 없으면 deny.  
snapshot이 없으면 allow하지 않는다.

정리:

OWL  
→ unknown 가능

SHACL / Policy / Safety Gate  
→ missing required data는 violation 또는 deny 조건

핵심 원칙:

Open-world reasoning과 runtime validation을 혼동하지 않는다.

## **10\. Solver 사용 원칙**

Solver는 중요하지만 runtime hot path에 들어가면 안 된다.

사용 가능한 경우:

Task allocation  
Route planning  
Resource optimization  
Schedule adjustment  
Conflict resolution proposal

Solver 결과는 곧바로 실행 명령이 아니다.

Solver result  
→ PlanCandidate  
→ Validation  
→ Policy Check  
→ Approval  
→ Safety Gate  
→ ExecutionRequest

핵심 원칙:

Solver는 candidate plan을 만든다.  
Solver는 physical execution authority를 가지지 않는다.

---

## **11\. AI Output과 Reasoning의 경계**

AI output은 reasoning result가 아니다.

SLM / LLM output  
→ interpretation  
→ candidate  
→ hypothesis

AI output은 다음이 될 수 없다.

Evidence  
ApprovedAction  
ExecutionRequest  
Safety Gate decision  
Ontology fact  
Policy decision

AI output이 ontology에 반영되려면 다음을 거쳐야 한다.

AI output  
→ Candidate  
→ Evidence Binding  
→ Validation  
→ Human or Policy Review  
→ Ontology / ABox update  
→ Audit

핵심 원칙:

AI는 후보를 만든다.  
Ontology와 validation layer가 후보를 검증한다.

## **12\. Anti-Patterns**

| Anti-pattern | 문제 | 대안 |
| ----- | ----- | ----- |
| OWL reasoner를 Safety Gate에서 실행 | runtime jitter 발생 | reasoning result materialization |
| SHACL을 Safety Gate hot path에서 full validation | 실행 지연 | shape result flag lookup |
| SPARQL query를 hot path에서 실행 | triple store latency 발생 | materialized relation map |
| Policy를 ontology axiom으로 대체 | 운영 정책 변경이 어려움 | Policy Engine 분리 |
| AI output을 Evidence로 사용 | hallucination이 사실처럼 취급됨 | evidence binding 필요 |
| Solver result를 바로 ExecutionRequest로 사용 | 검증/승인 우회 | PlanCandidate로 취급 |
| Missing data를 OWL로 invalid 처리하려 함 | Open World와 충돌 | SHACL 사용 |
| 모든 제약을 하나의 rule engine에 몰아넣음 | 책임 경계 붕괴 | reasoning type별 분리 |
| Runtime state를 TBox class로 표현 | 시간 변화 표현 실패 | State/Event/TimeInterval 사용 |
| Safety Gate에서 external API 호출 | network jitter 발생 | health state snapshot 사용 |

---

## **13\. 최종 운영 원칙**

Semantic reasoning belongs to OWL.  
Data validation belongs to SHACL.  
Graph relationship validation belongs to SPARQL / graph query.  
Operational permission belongs to Policy Engine.  
Optimization belongs to Solver.  
Runtime execution readiness belongs to Safety Gate.  
AI output is candidate interpretation.  
Safety Gate hot path must only lookup precomputed results.

---

## **14\. 최종 결론**

LEDO Reasoning and Constraint Model의 핵심은 reasoning을 많이 넣는 것이 아니다.

핵심은 reasoning의 책임을 정확히 나누고, 무거운 reasoning 결과를 runtime 전에 materialize하여, Safety Gate가 결정론적으로 조회할 수 있게 만드는 것이다.

최종 구조는 다음과 같다.

Ontology / OWL  
→ meaning and inference

SHACL  
→ data shape validation

SPARQL / Graph Query  
→ relationship validation

Policy Engine  
→ permission and approval logic

Solver  
→ optimization candidate

World State  
→ current runtime facts

Safety Gate Runtime Snapshot  
→ precomputed execution readiness view

Safety Gate  
→ deterministic final lookup

최종 원칙은 다음과 같다.

Reasoning creates meaning.  
Constraint validation filters invalid structure.  
Policy determines permission.  
Snapshot materializes runtime-ready facts.  
Safety Gate only reads and decides.
