# **Action Registry** 

## **1\. Overview**

`action_registry` defines and governs the controlled operational vocabulary of Action Types used in the LEDO Ontology-Centric Cyber-Physical System.

Its purpose is to prevent uncontrolled creation of actions and ensure that every `ActionCandidate`, `DecisionCase`, `ApprovedAction`, `ExecutionRequest`, and downstream execution-related object references a known, versioned, validated, and semantically grounded Action Type.

The Action Registry is not a list of command names. It is a **semantic and operational contract registry** that defines what an action means, what it may target, what evidence it requires, which policies must validate it, what approval level is required, which runtime validations must pass, and which external system boundary must handle physical execution.

---

## **2\. Core Principle**

An Action Type is a controlled operational vocabulary.

An `ActionCandidate` is not execution.

An `ApprovedAction` is not a physical command.

An `ExecutionRequest` is not a physical command.

An `ExternalControlRequest` is still not low-level physical control logic.

Physical execution belongs to external systems, such as robot fleet managers, PLCs, SCADA systems, safety-rated controllers, equipment controllers, or site platforms.

The Action Registry must preserve this boundary.

LLM / Agent  
    → proposes ActionCandidate

Ontology / Policy / Safety Gate  
    → validates whether the action may proceed

Action Registry  
    → defines whether the Action Type is known, valid, versioned, and allowed

Execution Request  
    → expresses approved operational intent

External System  
    → performs physical execution

---

## **3\. Design Position in the LEDO Architecture**

`action_registry` sits between semantic decision-making and execution preparation.

Domain Agent Layer  
        ↓  
ActionCandidate  
        ↓  
Decision Router / Escalation Layer  
        ↓  
Policy / Approval / Safety Gate  
        ↓  
ApprovedAction  
        ↓  
ExecutionRequest Builder  
        ↓  
action\_registry validation  
        ↓  
adapter\_registry compatibility check  
        ↓  
ExternalControlRequest  
        ↓  
External System

The Action Registry must be consulted before an action can move from candidate state into any validated operational lifecycle object.

---

## **4\. Purpose**

The Action Registry exists to ensure the following:

1. No unregistered Action Type can enter the system.  
2. Every Action Type has an explicit semantic meaning.  
3. Every Action Type has a declared target scope.  
4. Every Action Type declares required evidence.  
5. Every Action Type declares required policies.  
6. Every Action Type declares approval requirements.  
7. Every Action Type declares runtime validation requirements.  
8. Every Action Type declares whether external execution is possible.  
9. Every Action Type declares which adapter categories may support it.  
10. Every Action Type declares the physical execution boundary.  
11. Every Action Type has version, lifecycle status, owner, and migration metadata.  
12. Every Action Type can be audited, tested, and migrated.

---

## **5\. Core Distinctions**

### **5.1 ActionType**

`ActionType` is the registered abstract operational vocabulary.

Example:

DISPATCH\_ROBOT  
LOCK\_ZONE  
STOP\_WORK  
REQUEST\_INSPECTION  
NOTIFY\_MANAGER

It defines what kind of action is allowed to exist.

---

### **5.2 ActionCandidate**

`ActionCandidate` is a proposed action generated by an agent, rule engine, operator, or workflow.

It is not approved.

It is not executable.

It must reference a registered Action Type.

---

### **5.3 ApprovedAction**

`ApprovedAction` is an action that has passed policy, approval, and safety validation.

It is still not a physical command.

It means:

This action is approved for execution preparation.

---

### **5.4 ExecutionRequest**

`ExecutionRequest` is a structured operational request generated from an `ApprovedAction`.

It prepares execution intent for adapter routing.

It is still not low-level physical control.

---

### **5.5 ExternalControlRequest**

`ExternalControlRequest` is a request sent to an external system adapter.

It must remain at the high-level intent boundary.

It must not contain robot motion primitives, PLC ladder logic, raw actuator commands, or safety-rated controller logic.

## **7\. Non-Scope**

The Action Registry does not define the following:

1. Physical machine commands  
2. PLC commands  
3. SCADA internal commands  
4. Robot motion planning  
5. Robot joint control  
6. Emergency control logic inside safety-rated controllers  
7. Site-specific safety thresholds  
8. Domain-specific sensor thresholds  
9. Site-specific approval authority  
10. Low-level actuator behavior  
11. Vendor-specific command payloads  
12. External system retry logic  
13. Hardware interlock behavior

These belong to external systems, domain modules, policy engines, runtime validation modules, safety-rated controllers, robot middleware, PLCs, SCADA systems, or adapter implementations.

## **13\. Action Lifecycle Alignment**

Action Types may appear in the following lifecycle objects:

ActionCandidate  
    ↓  
DecisionCase  
    ↓  
Approval  
    ↓  
ApprovedAction  
    ↓  
SafetyGateResult  
    ↓  
ExecutionRequest  
    ↓  
ExternalControlRequest  
    ↓  
FeedbackEvent  
    ↓  
AuditRecord

The Action Type remains a reference throughout the lifecycle.

It must not become a physical command.

The same `action_type_id` must be traceable across all lifecycle objects.

---

## **14\. Lifecycle Object Rules**

### **14.1 ActionCandidate Rule**

An `ActionCandidate` is valid only if:

1. Its `action_type_id` exists in the Action Registry.  
2. The Action Type status is `active`.  
3. The proposed target type is allowed.  
4. Minimum evidence references are present or explicitly pending.  
5. The candidate source is recorded.  
6. The candidate is traceable to a domain agent, rule, user, or workflow.

Final rule:

No registered Action Type, no valid ActionCandidate.

---

### **14.2 DecisionCase Rule**

A `DecisionCase` is valid only if:

1. It references a valid `ActionCandidate`.  
2. It includes evidence evaluation.  
3. It includes policy lookup results.  
4. It includes escalation classification.  
5. It includes approval requirement mapping.

Final rule:

No valid ActionCandidate, no DecisionCase.

---

### **14.3 ApprovedAction Rule**

An `ApprovedAction` is valid only if:

1. Required approval is completed.  
2. Approval authority is valid.  
3. Required policy checks pass.  
4. Required evidence is bound.  
5. The action is not blocked or deprecated.  
6. Runtime validation requirements are declared.

Final rule:

No approval, no ApprovedAction.

---

### **14.4 ExecutionRequest Rule**

An `ExecutionRequest` is valid only if:

1. It references an `ApprovedAction`.  
2. Safety Gate has passed.  
3. Runtime validation has passed or is explicitly scheduled.  
4. Adapter compatibility is available.  
5. External system boundary is declared.  
6. Physical execution remains delegated to external systems.

Final rule:

No Safety Gate pass, no ExecutionRequest.

---

## **15\. Validation Rules**

An Action Type is valid only when all required conditions are satisfied.

Required conditions:

1. It exists in the Action Registry.  
2. Its status is `active`.  
3. Its target type is allowed.  
4. Its ontology class mapping is declared.  
5. Required evidence types are declared.  
6. Required policy references are declared.  
7. Required approval level is declared.  
8. Runtime validation references are declared.  
9. Preconditions are declared.  
10. Postconditions are declared.  
11. Safety boundary is explicit.  
12. Execution boundary is explicit.  
13. Supported adapters are declared when external execution is possible.  
14. External system type references are declared when external execution is possible.  
15. Version is valid.  
16. Owner module is declared.  
17. Migration metadata is declared if deprecated.

If any required condition is missing, the action must fail validation.

---

## **16\. Runtime Validation Requirements**

Every executable Action Type must declare runtime validation references.

Examples:

validation:worker\_not\_in\_hazard\_zone  
validation:robot\_available  
validation:equipment\_not\_locked\_out  
validation:zone\_accessible  
validation:sensor\_data\_fresh  
validation:external\_system\_reachable  
validation:approval\_not\_expired

Runtime validation is different from ontology definition.

The ontology defines meaning.

Runtime validation checks whether the current world state allows the action to proceed now.

---

## **17\. Preconditions, Postconditions, and Invariants**

### **17.1 Preconditions**

Preconditions define what must be true before an action can proceed.

Example:

A robot can be dispatched only if the robot is available and the target zone is accessible.

---

### **17.2 Postconditions**

Postconditions define what must be true after the action request is accepted or completed.

Example:

A dispatch request must create a mission tracking record.

---

### **17.3 Invariants**

Invariants define what must never be violated.

Example:

LEDO must not generate low-level robot motion primitives.

LEDO must not bypass safety-rated controllers.

Invariants are critical because they preserve the boundary between ontology-driven operational intent and physical control systems.

---

The Action Registry must reference and integrate with other registries.

| ----- | ----- |

| Policy references | policy\_registry |
| Approval requirements | approval\_registry |
| Runtime validation references | snapshot\_schema\_registry / runtime\_validation\_registry |
| Adapter references | adapter\_registry |
| External system references | external\_system\_registry |
| Target identity | identity\_registry |
| Ontology meaning | ontology\_registry / ontology\_foundation |
| Capability definitions | capability\_registry |
| Risk model | risk\_registry |
| Audit event model | audit\_event\_registry |
| Version migration | migration\_registry |

The Action Registry must not duplicate these registries. It should reference them.

---

## **19\. Relationship to adapter\_registry**

The Action Registry defines which adapters may support an Action Type.

The Adapter Registry determines which adapter is currently available and compatible.

action\_registry:  
    What adapter categories may support this Action Type?

adapter\_registry:  
    Which registered adapter instance can handle this ExecutionRequest now?

Example:

Action Type: DISPATCH\_ROBOT  
supported\_adapter\_refs:  
  \- adapter:robot\_fleet\_adapter  
  \- adapter:ros2\_robot\_adapter

At runtime:

ExecutionRequest  
    ↓  
Action Registry validates allowed adapter category  
    ↓  
Adapter Registry selects actual adapter instance

The Action Registry does not select live adapters.

Live adapter selection belongs to `adapter_registry`.

---

## **20\. Relationship to Ontology**

Every Action Type should have a semantic IRI.

Example:

action\_type\_id: DISPATCH\_ROBOT  
semantic\_iri: ledo:DispatchRobotAction

Ontology may define:

ledo:DispatchRobotAction  
    rdf:type ledo:ActionType ;  
    rdfs:subClassOf ledo:RobotAction ;  
    ledo:requiresCapability ledo:RobotDispatchCapability ;  
    ledo:hasAllowedTarget ledo:Robot ;  
    ledo:hasAllowedTarget ledo:WorkZone ;  
    ledo:requiresPolicy ledo:RobotDispatchPolicy .

The Action Registry is the operational registry.

The ontology is the semantic foundation.

They must be aligned but not collapsed into one object.

## **21\. Domain Extension Rule**

Domain Modules may propose new Action Types.

However, no domain-specific Action Type may be used until it is registered, reviewed, versioned, mapped, validated, and activated.

Recommended flow:

Domain Module  
    ↓  
Action Type Proposal  
    ↓  
Semantic Review  
    ↓  
Registry Review  
    ↓  
Policy Mapping  
    ↓  
Evidence Mapping  
    ↓  
Approval Mapping  
    ↓  
Runtime Validation Mapping  
    ↓  
Adapter Boundary Mapping  
    ↓  
Test Case Generation  
    ↓  
Activation

Domain Modules must not bypass the Action Registry.

## **22\. Versioning and Migration**

Every Action Type must be versioned.

Version changes are required when any of the following changes:

1. Allowed target types change.  
2. Required evidence changes.  
3. Required policy references change.  
4. Approval level changes.  
5. Runtime validation rules change.  
6. Adapter boundary changes.  
7. External system boundary changes.  
8. Safety boundary changes.  
9. Preconditions or postconditions change.  
10. Invariants change.

Status values:

draft  
active  
deprecated  
migration\_required  
retired  
blocked

A deprecated Action Type must declare:

deprecated\_since: datetime  
replacement\_action\_type\_id: string | null  
migration\_notes: string

A blocked Action Type must not be used in new lifecycle objects.

## **26\. Core Validation Function**

def validate\_action\_type\_for\_candidate(  
    entry: ActionRegistryEntry,  
    target\_type: str,  
) \-\> None:  
    if entry.status \!= ActionStatus.ACTIVE:  
        raise InvalidActionTypeError(  
            f"Action Type is not active: {entry.action\_type\_id}"  
        )

    if target\_type not in entry.allowed\_target\_types:  
        raise InvalidActionTargetError(  
            f"Target type '{target\_type}' is not allowed for "  
            f"Action Type '{entry.action\_type\_id}'"  
        )

    if not entry.required\_evidence\_types:  
        raise InvalidActionRegistryEntryError(  
            "required\_evidence\_types must be declared"  
        )

    if not entry.required\_policy\_refs:  
        raise InvalidActionRegistryEntryError(  
            "required\_policy\_refs must be declared"  
        )

    if not entry.required\_approval\_level:  
        raise InvalidActionRegistryEntryError(  
            "required\_approval\_level must be declared"  
        )

    if not entry.runtime\_validation\_refs:  
        raise InvalidActionRegistryEntryError(  
            "runtime\_validation\_refs must be declared"  
        )

    if not entry.execution\_boundary:  
        raise InvalidActionRegistryEntryError(  
            "execution\_boundary must be declared"  
        )

    if not entry.safety\_boundary:  
        raise InvalidActionRegistryEntryError(  
            "safety\_boundary must be declared"  
        )

3\. Reject deprecated Action Type without migration.  
4\. Reject blocked Action Type.  
5\. Reject target type not allowed by Action Registry.  
6\. Reject missing evidence requirements.  
7\. Reject missing policy references.  
8\. Reject missing approval level.  
9\. Reject missing runtime validation references.  
10\. Reject missing execution boundary.  
11\. Reject missing safety boundary.  
12\. Validate adapter compatibility reference.  
13\. Validate external system boundary reference.  
14\. Validate version migration rule.  
15\. Validate that ExecutionRequest cannot be created without Safety Gate pass.

---

## **28\. Final Rule**

No registered Action Type,  
no valid ActionCandidate.

No valid ActionCandidate,  
no DecisionCase.

No valid DecisionCase,  
no Approval.

No approval,  
no ApprovedAction.

No Safety Gate pass,  
no ExecutionRequest.

No adapter compatibility,  
no ExternalControlRequest.

No External System,  
no physical execution.

The Action Registry is therefore the first deterministic control point in the LEDO action lifecycle.

It prevents uncontrolled action creation, protects the boundary between intent and execution, and ensures that every action is semantically grounded, policy-bound, approval-aware, runtime-validatable, adapter-compatible, and auditable.

**Action Registry** 

## **1\. 개요**

`action_registry`는 LEDO Ontology-Centric Cyber-Physical System에서 사용되는 Action Type의 통제된 운영 어휘를 정의하고 관리하는 핵심 레지스트리이다.

이 모듈의 목적은 시스템 내부에서 무분별하게 Action이 생성되는 것을 방지하고, 모든 `ActionCandidate`, `DecisionCase`, `ApprovedAction`, `ExecutionRequest`, 그리고 이후 실행 관련 객체들이 반드시 알려져 있고, 버전 관리되며, 검증된 Action Type을 참조하도록 보장하는 것이다.

Action Registry는 단순한 명령어 목록이 아니다.  
Action Registry는 **하나의 Action이 무엇을 의미하는지, 어떤 대상에 적용될 수 있는지, 어떤 증거가 필요한지, 어떤 정책을 통과해야 하는지, 어떤 승인 수준이 필요한지, 어떤 런타임 검증을 통과해야 하는지, 그리고 물리 실행은 어떤 외부 시스템 경계에서 처리되어야 하는지를 정의하는 의미론적·운영적 계약 레지스트리**이다.

---

## **2\. 핵심 원칙**

Action Type은 통제된 운영 어휘이다.

`ActionCandidate`는 실행이 아니다.

`ApprovedAction`은 물리 명령이 아니다.

`ExecutionRequest`도 물리 명령이 아니다.

`ExternalControlRequest` 또한 저수준 물리 제어 로직이 아니다.

물리적 실행은 로봇 Fleet Manager, PLC, SCADA, Safety-rated Controller, 장비 제어기, 현장 플랫폼과 같은 외부 시스템의 책임이다.

Action Registry는 이 경계를 반드시 보존해야 한다.

LLM / Agent

    → ActionCandidate 제안

Ontology / Policy / Safety Gate

    → 해당 Action이 진행 가능한지 검증

Action Registry

    → 해당 Action Type이 알려져 있고, 유효하며, 버전 관리되고, 허용되는지 검증

ExecutionRequest

    → 승인된 운영 의도를 표현

External System

    → 실제 물리 실행 수행

## **4\. 목적**

Action Registry는 다음을 보장하기 위해 존재한다.

1. 등록되지 않은 Action Type은 시스템에 진입할 수 없다.  
2. 모든 Action Type은 명확한 의미를 가진다.  
3. 모든 Action Type은 허용 가능한 대상 범위를 가진다.  
4. 모든 Action Type은 필요한 증거를 선언한다.  
5. 모든 Action Type은 필요한 정책을 선언한다.  
6. 모든 Action Type은 필요한 승인 수준을 선언한다.  
7. 모든 Action Type은 필요한 런타임 검증을 선언한다.  
8. 모든 Action Type은 외부 실행 가능 여부를 선언한다.  
9. 모든 Action Type은 어떤 adapter category가 지원 가능한지 선언한다.  
10. 모든 Action Type은 물리 실행 경계를 선언한다.  
11. 모든 Action Type은 version, lifecycle status, owner, migration metadata를 가진다.  
12. 모든 Action Type은 감사, 테스트, 마이그레이션이 가능해야 한다.

## **5\. 핵심 구분**

### **5.1 ActionType**

ActionType은 시스템 안에서 어떤 종류의 Action이 존재할 수 있는지를 정의한다.

---

### **5.2 ActionCandidate**

`ActionCandidate`는 agent, rule engine, operator, workflow 등이 제안한 Action이다.

아직 승인되지 않았다.

아직 실행 가능하지 않다.

반드시 등록된 Action Type을 참조해야 한다.

---

### **5.3 ApprovedAction**

`ApprovedAction`은 policy, approval, safety validation을 통과한 Action이다.

그러나 여전히 물리 명령은 아니다.

그 의미는 다음과 같다.

이 Action은 실행 준비 단계로 넘어갈 수 있도록 승인되었다.

---

### **5.4 ExecutionRequest**

`ExecutionRequest`는 `ApprovedAction`으로부터 생성된 구조화된 운영 요청이다.

adapter routing을 위해 실행 의도를 준비한다.

그러나 여전히 저수준 물리 제어 명령은 아니다.

---

### **5.5 ExternalControlRequest**

`ExternalControlRequest`는 외부 시스템 adapter로 전달되는 요청이다.

이 요청 역시 고수준 의도 경계를 유지해야 한다.

여기에는 robot motion primitive, PLC ladder logic, raw actuator command, safety-rated controller logic 등이 포함되면 안 된다.

## **14\. Lifecycle Object 규칙**

### **14.1 ActionCandidate 규칙**

`ActionCandidate`는 다음 조건을 만족할 때만 유효하다.

1. `action_type_id`가 Action Registry에 존재한다.  
2. 해당 Action Type의 status가 `active`이다.  
3. 제안된 target type이 허용되어 있다.  
4. 최소 evidence reference가 존재하거나 명시적으로 pending 처리되어 있다.  
5. candidate source가 기록되어 있다.  
6. candidate가 domain agent, rule, user, workflow 중 어디서 발생했는지 추적 가능하다.

최종 규칙:

등록된 Action Type이 없으면 유효한 ActionCandidate도 없다.

---

### **14.2 DecisionCase 규칙**

`DecisionCase`는 다음 조건을 만족할 때만 유효하다.

1. 유효한 `ActionCandidate`를 참조한다.  
2. evidence evaluation을 포함한다.  
3. policy lookup result를 포함한다.  
4. escalation classification을 포함한다.  
5. approval requirement mapping을 포함한다.

최종 규칙:

유효한 ActionCandidate가 없으면 DecisionCase도 없다.

---

### **14.3 ApprovedAction 규칙**

`ApprovedAction`은 다음 조건을 만족할 때만 유효하다.

1. 필요한 approval이 완료되어 있다.  
2. approval authority가 유효하다.  
3. 필요한 policy check가 통과되어 있다.  
4. 필요한 evidence가 binding되어 있다.  
5. 해당 Action이 blocked 또는 deprecated 상태가 아니다.  
6. runtime validation requirement가 선언되어 있다.

최종 규칙:

승인이 없으면 ApprovedAction도 없다.

---

### **14.4 ExecutionRequest 규칙**

`ExecutionRequest`는 다음 조건을 만족할 때만 유효하다.

1. `ApprovedAction`을 참조한다.  
2. Safety Gate를 통과했다.  
3. runtime validation을 통과했거나 명시적으로 scheduled 되어 있다.  
4. adapter compatibility가 존재한다.  
5. external system boundary가 선언되어 있다.  
6. physical execution은 여전히 외부 시스템에 위임되어 있다.

최종 규칙:

Safety Gate 통과가 없으면 ExecutionRequest도 없다.

---

## **15\. Validation Rules**

Action Type은 다음 조건을 모두 만족할 때만 유효하다.

필수 조건:

1. Action Registry에 존재해야 한다.  
2. status가 `active`이어야 한다.  
3. target type이 허용되어야 한다.  
4. ontology class mapping이 선언되어야 한다.  
5. required evidence type이 선언되어야 한다.  
6. required policy reference가 선언되어야 한다.  
7. required approval level이 선언되어야 한다.  
8. runtime validation reference가 선언되어야 한다.  
9. precondition이 선언되어야 한다.  
10. postcondition이 선언되어야 한다.  
11. safety boundary가 명시되어야 한다.  
12. execution boundary가 명시되어야 한다.  
13. 외부 실행이 가능한 경우 supported adapter가 선언되어야 한다.  
14. 외부 실행이 가능한 경우 external system type reference가 선언되어야 한다.  
15. version이 유효해야 한다.  
16. owner module이 선언되어야 한다.  
17. deprecated 상태라면 migration metadata가 선언되어야 한다.

필수 조건 중 하나라도 누락되면 해당 Action은 validation에 실패해야 한다.

---

## **16\. Runtime Validation Requirements**

실행 가능한 모든 Action Type은 runtime validation reference를 선언해야 한다.

예시:

validation:worker\_not\_in\_hazard\_zone

validation:robot\_available

validation:equipment\_not\_locked\_out

validation:zone\_accessible

validation:sensor\_data\_fresh

validation:external\_system\_reachable

validation:approval\_not\_expired

Runtime validation은 ontology definition과 다르다.

Ontology는 의미를 정의한다.

Runtime validation은 현재 world state 기준으로 지금 이 Action이 진행 가능한지를 확인한다.

## **17\. Preconditions, Postconditions, Invariants**

### **17.1 Preconditions**

Precondition은 Action이 진행되기 전에 반드시 참이어야 하는 조건을 정의한다.

예시:

로봇은 사용 가능 상태이고, target zone이 접근 가능할 때만 dispatch될 수 있다.

---

### **17.2 Postconditions**

Postcondition은 Action request가 accepted 또는 completed 된 후 반드시 참이어야 하는 조건을 정의한다.

예시:

dispatch request는 mission tracking record를 생성해야 한다.

---

### **17.3 Invariants**

Invariant는 어떤 경우에도 위반되면 안 되는 조건을 정의한다.

예시:

LEDO는 저수준 robot motion primitive를 생성하면 안 된다.

LEDO는 safety-rated controller를 우회하면 안 된다.

Invariant는 ontology-driven operational intent와 physical control system 사이의 경계를 보존하기 때문에 매우 중요하다.

---

## **18\. 다른 Registry와의 관계**

Action Registry는 다른 registry들과 참조 관계를 가져야 한다.

| Reference Type | Registry |
| ----- | ----- |
| Evidence requirements | evidence\_registry |
| Policy references | policy\_registry |
| Approval requirements | approval\_registry |
| Runtime validation references | snapshot\_schema\_registry / runtime\_validation\_registry |
| Adapter references | adapter\_registry |
| External system references | external\_system\_registry |
| Target identity | identity\_registry |
| Ontology meaning | ontology\_registry / ontology\_foundation |
| Capability definitions | capability\_registry |
| Risk model | risk\_registry |
| Audit event model | audit\_event\_registry |
| Version migration | migration\_registry |

Action Registry는 이 registry들을 중복해서 정의하면 안 된다.  
Action Registry는 이들을 참조해야 한다.

---

## **19\. adapter\_registry와의 관계**

Action Registry는 어떤 adapter가 해당 Action Type을 지원할 수 있는지 정의한다.

Adapter Registry는 현재 어떤 adapter instance가 사용 가능하고 호환되는지 결정한다.

action\_registry:

    이 Action Type은 어떤 adapter category가 지원할 수 있는가?

adapter\_registry:

    지금 이 ExecutionRequest를 처리할 수 있는 실제 adapter instance는 무엇인가?

예시:

Action Type: DISPATCH\_ROBOT

supported\_adapter\_refs:

  \- adapter:robot\_fleet\_adapter

  \- adapter:ros2\_robot\_adapter

Runtime에서는 다음과 같이 진행된다.

ExecutionRequest

    ↓

Action Registry validates allowed adapter category

    ↓

Adapter Registry selects actual adapter instance

Action Registry는 live adapter를 선택하지 않는다.

실시간 adapter 선택은 `adapter_registry`의 책임이다.

---

## **20\. Ontology와의 관계**

모든 Action Type은 semantic IRI를 가져야 한다.

예시:

action\_type\_id: DISPATCH\_ROBOT

semantic\_iri: ledo:DispatchRobotAction

Ontology에서는 다음과 같이 정의할 수 있다.

ledo:DispatchRobotAction

    rdf:type ledo:ActionType ;

    rdfs:subClassOf ledo:RobotAction ;

    ledo:requiresCapability ledo:RobotDispatchCapability ;

    ledo:hasAllowedTarget ledo:Robot ;

    ledo:hasAllowedTarget ledo:WorkZone ;

    ledo:requiresPolicy ledo:RobotDispatchPolicy .

Action Registry는 운영 레지스트리이다.

Ontology는 의미론적 기반이다.

둘은 정렬되어야 하지만 하나의 객체로 합쳐지면 안 된다.

---

## **21\. Domain Extension Rule**

Domain Module은 새로운 Action Type을 제안할 수 있다.

그러나 domain-specific Action Type은 등록, 검토, 버전 관리, 매핑, 검증, 활성화 과정을 거치기 전까지 사용할 수 없다.

## **22\. Versioning 및 Migration**

모든 Action Type은 versioning되어야 한다.

다음 항목 중 하나라도 변경되면 version 변경이 필요하다.

1. allowed target type 변경  
2. required evidence 변경  
3. required policy reference 변경  
4. approval level 변경  
5. runtime validation rule 변경  
6. adapter boundary 변경  
7. external system boundary 변경  
8. safety boundary 변경  
9. precondition 또는 postcondition 변경  
10. invariant 변경

Status 값:

draft

active

deprecated

migration\_required

retired

blocked

Deprecated Action Type은 다음을 선언해야 한다.

deprecated\_since: datetime

replacement\_action\_type\_id: string | null

migration\_notes: string

Blocked Action Type은 새로운 lifecycle object에서 사용되면 안 된다.

## **26\. Core Validation Function**

def validate\_action\_type\_for\_candidate(

    entry: ActionRegistryEntry,

    target\_type: str,

) \-\> None:

    if entry.status \!= ActionStatus.ACTIVE:

        raise InvalidActionTypeError(

            f"Action Type is not active: {entry.action\_type\_id}"

        )

    if target\_type not in entry.allowed\_target\_types:

        raise InvalidActionTargetError(

            f"Target type '{target\_type}' is not allowed for "

            f"Action Type '{entry.action\_type\_id}'"

        )

    if not entry.required\_evidence\_types:

        raise InvalidActionRegistryEntryError(

            "required\_evidence\_types must be declared"

        )

    if not entry.required\_policy\_refs:

        raise InvalidActionRegistryEntryError(

            "required\_policy\_refs must be declared"

        )

    if not entry.required\_approval\_level:

        raise InvalidActionRegistryEntryError(

            "required\_approval\_level must be declared"

        )

    if not entry.runtime\_validation\_refs:

        raise InvalidActionRegistryEntryError(

            "runtime\_validation\_refs must be declared"

        )

    if not entry.execution\_boundary:

        raise InvalidActionRegistryEntryError(

            "execution\_boundary must be declared"

        )

    if not entry.safety\_boundary:

        raise InvalidActionRegistryEntryError(

            "safety\_boundary must be declared"

        )

## **28\. Final Rule**

등록된 Action Type이 없으면,

유효한 ActionCandidate도 없다.

유효한 ActionCandidate가 없으면,

DecisionCase도 없다.

유효한 DecisionCase가 없으면,

Approval도 없다.

Approval이 없으면,

ApprovedAction도 없다.

Safety Gate 통과가 없으면,

ExecutionRequest도 없다.

Adapter compatibility가 없으면,

ExternalControlRequest도 없다.

External System이 없으면,

물리 실행도 없다.

Action Registry는 LEDO Action Lifecycle의 첫 번째 결정론적 통제 지점이다.

이 모듈은 통제되지 않은 Action 생성을 방지하고, intent와 execution 사이의 경계를 보호하며, 모든 Action이 의미론적으로 grounding되고, policy에 묶이고, approval을 인식하고, runtime validation이 가능하며, adapter와 호환되고, audit 가능한 상태가 되도록 보장한다.
