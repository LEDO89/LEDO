# **Decisionregistry**

## **1\. Overview**

`decision_registry` is a core registry in the LEDO Ontology-Centric Cyber-Physical System. It defines and governs Decision Types, Decision Case Types, Decision Criteria, Escalation Rules, Risk Classification, Evidence Evaluation Requirements, Policy Evaluation Requirements, and Approval Routing Requirements.

The purpose of this registry is to prevent an `ActionCandidate` generated by an agent from moving directly into approval or execution stages, and to ensure that every candidate action is reviewed through a registered Decision Rule and Decision Case structure.

`decision_registry` is not a simple list of decisions.

It is an **operational contract registry for judgment procedures and decision flows** that defines the following:

Which DecisionCase should an ActionCandidate be converted into?

Which evidence must be evaluated?

Which policies must be reviewed?

Which risk class should be assigned?

When is escalation required?

Which approval route should be used?

When should the case be automatically rejected or held?

Which lifecycle object may follow the Decision outcome?

In other words, `decision_registry` deterministically controls the judgment flow before an agent proposal can move toward human approval or execution preparation.

---

## **2\. Core Principle**

A Decision is not execution.

A Decision is not approval.

A Decision is not the Safety Gate.

A Decision is not a physical command.

A Decision is not adapter selection.

The role of a Decision is:

Convert an ActionCandidate into an evaluable DecisionCase,

evaluate evidence, policy, risk, priority, and approval requirements,

and route it to the next lifecycle stage.

The core principles of the Decision Registry are:

No valid Decision Rule,

no valid DecisionCase.

No valid DecisionCase,

no ApprovalRequest.

No ApprovalRequest,

no ApprovedAction.

No Safety Gate pass,

no ExecutionRequest.

The Decision Registry does not decide what to physically execute.  
It defines **which judgment procedure must be applied to a proposed Action**.

## **4\. Purpose**

The purpose of `decision_registry` is to ensure the following:

1. Define rules for converting ActionCandidate into DecisionCase  
2. Define Decision Types and Decision Case Types  
3. Define judgment criteria by Action Type  
4. Define Risk Class and Priority criteria  
5. Define evidence evaluation requirements  
6. Define policy evaluation requirements  
7. Define escalation rules  
8. Define approval routing rules  
9. Define conditions for auto-approval, hold, reject, or escalation  
10. Define lifecycle paths available to Decision outcomes  
11. Define audit rules for Decision results  
12. Prevent agents or LLMs from arbitrarily creating DecisionCases  
13. Prevent ApprovalRequest creation without a valid Decision

## **5\. Core Distinctions**

### **5.1 Decision Type**

`Decision Type` is a controlled judgment type allowed in the system.

Examples:

safety\_risk\_decision

robot\_dispatch\_decision

inspection\_decision

equipment\_status\_decision

zone\_access\_decision

compliance\_decision

resource\_allocation\_decision

workflow\_routing\_decision

emergency\_escalation\_decision

Decision Type defines what kinds of judgment procedures may exist.

---

### **5.2 Decision Rule**

`Decision Rule` defines how a specific ActionCandidate should be evaluated, which DecisionCase it should become, and which next lifecycle stage it should be routed to.

Examples:

A STOP\_WORK candidate follows the safety\_risk\_decision rule.

A DISPATCH\_ROBOT candidate follows the robot\_dispatch\_decision rule.

A REQUEST\_INSPECTION candidate follows the inspection\_decision rule.

---

### **5.3 DecisionCase**

`DecisionCase` is the official judgment object created for an ActionCandidate.

A DecisionCase includes the following:

action\_candidate\_id

action\_type\_id

decision\_rule\_id

target\_scope

evidence\_bundle\_refs

policy\_evaluation\_refs

risk\_class

priority\_level

escalation\_level

recommended\_approval\_level

decision\_outcome

DecisionCase is a core lifecycle object before approval.

---

### **5.4 Decision Criteria**

`Decision Criteria` are the criteria used for judgment.

Examples:

risk\_score

confidence\_score

evidence\_completeness

policy\_pass\_status

world\_state\_freshness

target\_scope\_validity

human\_presence\_risk

equipment\_status

robot\_availability

zone\_accessibility

Decision Criteria must be explicitly defined.

---

### **5.5 Decision Outcome**

`Decision Outcome` is the judgment result of a DecisionCase.

Recommended outcomes:

proceed\_to\_approval

auto\_reject

hold\_for\_more\_evidence

escalate

defer

merge\_with\_existing\_case

requires\_manual\_review

requires\_runtime\_revalidation

blocked

Decision Outcome is not execution.  
Decision Outcome is the routing result to the next lifecycle stage.

## **9\. Decision Outcome Model**

Recommended Decision Outcomes are:

proceed\_to\_approval

auto\_reject

hold\_for\_more\_evidence

escalate

defer

merge\_with\_existing\_case

requires\_manual\_review

requires\_runtime\_revalidation

blocked

### **9.1 proceed\_to\_approval**

The DecisionCase may move to the approval stage.

DecisionCase → ApprovalRequest

---

### **9.2 auto\_reject**

The case is automatically rejected because required conditions are not satisfied.

Examples:

Action Type is blocked

Target type mismatch

Required evidence missing

Policy hard-fail

---

### **9.3 hold\_for\_more\_evidence**

The case is held because evidence is insufficient.

DecisionCase remains open, but ApprovalRequest is not created.

---

### **9.4 escalate**

The case is routed to a higher-level judgment path due to high risk, uncertainty, or impact.

DecisionCase → Escalation Queue

---

### **9.5 defer**

Judgment is deferred when current world state is insufficient or a system dependency is unavailable.

---

### **9.6 merge\_with\_existing\_case**

The case is merged when an identical or similar DecisionCase already exists.

---

### **9.7 requires\_manual\_review**

The case must be reviewed by a human because automatic judgment is not sufficient.

---

### **9.8 requires\_runtime\_revalidation**

Additional validation is required because the runtime world state is changing rapidly.

---

### **9.9 blocked**

The case cannot proceed by rule.

Examples:

Action Type blocked

Agent Type unauthorized

Policy hard prohibition

Safety boundary violation

---

## **14\. Validation Rules**

A Decision Rule is valid only when the following conditions are satisfied:

1. `decision_rule_id` exists in the registry.  
2. Its status is `active`.  
3. Applicable Action Types are declared.  
4. Applicable Agent Types are declared.  
5. Applicable target types are declared.  
6. Required evidence references are declared.  
7. Required policy references are declared.  
8. Required runtime snapshot references are declared.  
9. Decision criteria are declared.  
10. Risk classification rule is declared.  
11. Priority classification rule is declared.  
12. Allowed decision outcomes are declared.  
13. Default decision outcome is declared.  
14. Decision boundary is declared.  
15. Approval boundary is declared.  
16. Execution boundary is declared.  
17. Safety boundary is declared.  
18. Audit event references are declared.  
19. Owner module is declared.  
20. Version is valid.  
21. If deprecated, migration metadata exists.

If any of these conditions are missing, the Decision Rule must not be used in the operational lifecycle.

---

## **15\. DecisionCase Creation Validation**

A DecisionCase may be created only when the following conditions are satisfied:

Is the ActionCandidate valid?

Does the Action Type exist in action\_registry with active status?

Is the Agent Type that created the ActionCandidate authorized to propose this Action Type?

Is the Decision Rule applicable to this Action Type?

Is the Target Type included in applicable target types?

Does required Evidence exist or is it explicitly marked as pending?

Are required Policy references connected?

Does required Runtime Snapshot exist?

Does it comply with the decision boundary?

If these conditions are not satisfied, a DecisionCase must not be created.

---

## **16\. Evidence Evaluation Rule**

Decision Registry declares which evidence is required.

However, it does not generate the evidence itself.

decision\_registry:

    Declares which evidence is required

evidence\_registry:

    Defines evidence types and schemas

evidence\_binder:

    Binds actual evidence to ActionCandidate / DecisionCase

If required evidence is missing, the DecisionCase must result in one of the following outcomes:

hold\_for\_more\_evidence

auto\_reject

requires\_manual\_review

The selected outcome is defined by the Decision Rule.

---

## **17\. Policy Evaluation Rule**

Decision Registry declares which policy evaluations are required.

However, the actual policy pass/fail logic belongs to the Policy Registry or Policy Engine.

decision\_registry:

    Declares which policies are required

policy\_registry:

    Manages policy definitions

policy\_engine:

    Evaluates pass / fail / warning / requires\_review

When a policy hard-fail occurs, the typical outcomes are:

auto\_reject

blocked

escalate

---

## **18\. Risk Classification Rule**

DecisionCase must have a risk class.

Risk Classification may consider the following factors:

hazard severity

worker exposure

equipment state

zone condition

robot proximity

sensor confidence

policy warning

historical incident pattern

runtime uncertainty

Risk classes are the canonical `RiskLevel` set defined in `03_core_specifications/07_decision_approval_matrix/07_decision_approval_matrix.md` Section 8 ("Risk Level"):

INFO

NOTICE

WARNING

HIGH\_RISK

CRITICAL\_EMERGENCY

EXCEPTIONAL

Decision Registry references which risk classification rule should be used.

The actual risk score calculation may be handled by a risk engine or domain module.

## **20\. Escalation Rule**

Decision Registry defines when escalation is required.

Example escalation conditions:

risk\_class \>= CRITICAL\_EMERGENCY

evidence conflict detected

policy warning unresolved

multiple agents disagree

human proximity risk high

external system unavailable

approval timeout

repeated failed attempts

Example escalation targets:

site\_supervisor

safety\_supervisor

site\_manager

war\_room

emergency\_controller

compliance\_officer

Escalation is not approval.  
Escalation routes the case to a higher judgment or approval path.

## **21\. Approval Routing Rule**

Decision Registry defines which approval route should be used when the Decision Outcome is `proceed_to_approval`.

DecisionCase

    →
approval\_routing\_rule\_ref

    →
ApprovalRequest

    →
approval\_registry

Decision Registry does not perform approval directly.

Approval is handled by `approval_registry` and human authority.

## **22\. Relationship to Agent Vocabulary Registry**

`agent_vocabulary_registry` controls which Agent Type may propose which Action Type.

`decision_registry` controls which Decision Rule the resulting ActionCandidate must follow.

agent\_vocabulary\_registry:

    Is this agent allowed to propose this ActionCandidate?

decision\_registry:

    Which judgment procedure must this ActionCandidate follow?

Even if an agent has a high confidence score, it must not bypass the Decision Registry.

**Implementation status note:** `agent_vocabulary_registry` is out of scope for the current implementation stage (see `06_registry_specs/README.md` Section 11 and `07_implementation_plan/implementation_slice_2/implementation_slice_2_plan.md`). Until it exists, `applicable_agent_type_refs` (Section 10) must be treated as an unvalidated placeholder field: it may be populated with plain strings for documentation purposes, but the cross-registry reference resolver (`06_registry_specs/README.md` Section 12B) must not attempt to resolve or enforce it against a target registry. This is a temporary gap, not a design decision — when `agent_vocabulary_registry` is implemented, `applicable_agent_type_refs` must be brought under the same referential-integrity rule as every other `*_ref` / `*_refs` field.

---

## **23\. Relationship to Action Registry**

`action_registry` controls the existence and operational contract of Action Types.

`decision_registry` controls which judgment procedure an Action Type candidate must follow.

action\_registry:

    Is this Action Type valid?

decision\_registry:

    Which DecisionCase should a candidate of this Action Type become?

An Action Type that does not exist in the Action Registry cannot be converted into a DecisionCase.

---

## **24\. Relationship to Approval Registry**

`decision_registry` may determine the approval route.

However, the approval rule itself is managed by `approval_registry`.

decision\_registry:

    Does this DecisionCase require approval?

    If so, which approval route should it follow?

approval\_registry:

    Who can approve it, under what conditions and scope?

---

## **25\. Relationship to Policy Registry**

Decision Registry requires policy references.

Policy Registry manages policy definitions.

Policy Engine evaluates policy pass/fail.

decision\_registry:

    Declares which policy evaluations are required

policy\_registry:

    Manages policy meaning and rules

policy\_engine:

    Performs actual evaluation

---

## **26\. Relationship to Safety Gate**

Decision Registry does not replace the Safety Gate.

Decision occurs before approval.

The Safety Gate performs final execution validation before the ExecutionRequest.

Decision Registry:

    Evaluates ActionCandidate as a DecisionCase

    and determines approval routing

Safety Gate:

    Finally verifies whether an ApprovedAction

    is safe to prepare for execution now

Core principle:

Decision pass → Approval pass

Approval pass → RuntimeValidationResult → SafetyGatePass or SafetyGateBlock

---

## **27\. Relationship to Adapter Registry**

Decision Registry does not select adapters.

However, a Decision Rule may indicate whether the ActionCandidate may eventually follow an external execution path and whether an adapter compatibility check will be required later.

decision\_registry:

    Determines whether this ActionCandidate may have an external execution path

adapter\_registry:

    Selects the actual adapter instance after the Safety Gate

---

## **28\. Relationship to Ontology**

Every Decision Rule may have a semantic IRI.

Example:

decision\_rule\_id: decision:stop\_work\_safety\_risk\_v1

semantic\_iri: ledo:StopWorkSafetyRiskDecisionRule

In the ontology, it may be defined as follows:

ledo:StopWorkSafetyRiskDecisionRule

    rdf:type ledo:DecisionRule ;

    ledo:appliesToAction ledo:StopWorkAction ;

    ledo:requiresEvidence ledo:HazardDetectionSnapshot ;

    ledo:requiresPolicy ledo:StopWorkPolicy ;

    ledo:routesToApproval ledo:SafetyManagerApproval .

Ontology provides the semantic foundation of Decision Rules.

Decision Registry manages this foundation in the operational system through version, status, validation, routing, and audit rules.

## **29\. Versioning and Migration**

Decision Rules must be versioned.

A version change is required when any of the following changes:

1. Applicable Action Type  
2. Applicable Agent Type  
3. Applicable target type  
4. Required evidence  
5. Required policy reference  
6. Runtime snapshot requirement  
7. Decision criteria  
8. Risk classification rule  
9. Priority classification rule  
10. Escalation rule  
11. Approval routing rule  
12. Allowed decision outcome  
13. Decision / approval / execution / safety boundary

Status values:

draft

active

deprecated

migration\_required

retired

blocked

A deprecated Decision Rule must declare:

deprecated\_since: datetime

replacement\_decision\_rule\_id: string | null

migration\_notes: string

A blocked Decision Rule must not be used to create new DecisionCases.

## **33\. Core Validation Function**

def validate\_decision\_rule\_for\_candidate(

    entry: DecisionRegistryEntry,

    action\_type\_ref: str,

    agent\_type\_ref: str,

    target\_type: str,

) \-\> None:

    if entry.status \!= DecisionStatus.ACTIVE:

        raise InvalidDecisionRuleError(

            f"Decision Rule is not active: {entry.decision\_rule\_id}"

        )

    if action\_type\_ref not in entry.applicable\_action\_type\_refs:

        raise DecisionActionTypeMismatchError(

            f"Action Type '{action\_type\_ref}' is not applicable to "

            f"Decision Rule '{entry.decision\_rule\_id}'"

        )

    if agent\_type\_ref not in entry.applicable\_agent\_type\_refs:

        raise DecisionAgentTypeMismatchError(

            f"Agent Type '{agent\_type\_ref}' is not applicable to "

            f"Decision Rule '{entry.decision\_rule\_id}'"

        )

    if target\_type not in entry.applicable\_target\_types:

        raise DecisionTargetTypeMismatchError(

            f"Target Type '{target\_type}' is not applicable to "

            f"Decision Rule '{entry.decision\_rule\_id}'"

        )

    if not entry.required\_evidence\_refs:

        raise InvalidDecisionRuleError(

            "required\_evidence\_refs must be declared"

        )

    if not entry.required\_policy\_refs:

        raise InvalidDecisionRuleError(

            "required\_policy\_refs must be declared"

        )

    if not entry.decision\_criteria\_refs:

        raise InvalidDecisionRuleError(

            "decision\_criteria\_refs must be declared"

        )

    if not entry.allowed\_decision\_outcomes:

        raise InvalidDecisionRuleError(

            "allowed\_decision\_outcomes must be declared"

        )

    if entry.default\_decision\_outcome not in entry.allowed\_decision\_outcomes:

        raise InvalidDecisionRuleError(

            "default\_decision\_outcome must be included in allowed\_decision\_outcomes"

        )

    if not entry.decision\_boundary:

        raise InvalidDecisionRuleError(

            "decision\_boundary must be declared"

        )

    if not entry.approval\_boundary:

        raise InvalidDecisionRuleError(

            "approval\_boundary must be declared"

        )

    if not entry.execution\_boundary:

        raise InvalidDecisionRuleError(

            "execution\_boundary must be declared"

        )

    if not entry.safety\_boundary:

        raise InvalidDecisionRuleError(

            "safety\_boundary must be declared"

        )

## **35\. Final Rule**

No registered Decision Rule,

no valid DecisionCase.

No valid DecisionCase,

no ApprovalRequest.

If the Decision Outcome is not proceed\_to\_approval,

it cannot move to ApprovalRequest.

Decision is not Approval.

Decision is not the Safety Gate.

Decision is not ExecutionRequest.

Decision is not Physical Command.

Decision is not External System control.

`decision_registry` is a core deterministic judgment registry that controls the space between agent proposals and human approval in the LEDO Action Lifecycle.

This module prevents ActionCandidates from arbitrarily moving into the approval stage and ensures that every candidate action passes through evidence, policy, risk, priority, escalation, and approval routing criteria.

The core definition is:

Decision Registry

\= not a list of decision results,

but an operational contract registry that controls

the judgment criteria, evidence requirements, policy requirements,

risk classification, escalation, and approval routing

required for an ActionCandidate to become a DecisionCase

and move toward ApprovalRequest.
